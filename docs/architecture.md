# Архитектура

Проект построен на принципах микросервисной архитектуры и состоит из следующих ключевых компонентов:

- **`api`**: Веб-сервис на **FastAPI**, который принимает HTTP-запросы на загрузку изображений. Его основная задача — валидировать запрос, создать запись о задаче (job) в базе данных и передать саму задачу в очередь Celery.

- **`worker`**: Воркер **Celery**, который слушает очередь задач. Он выполняет ресурсоемкие операции: изменение размера изображений, конвертацию в формат `.webp` и загрузку в S3-хранилище.

- **`db`**: База данных **PostgreSQL** для хранения информации о задачах (jobs) и их статусах. Это позволяет отслеживать выполнение и получать результаты.

- **`redis`**: Брокер сообщений для Celery. API отправляет задачи в Redis, а воркеры забирают их оттуда для выполнения. Redis также используется для хранения результатов задач.

- **`minio`**: S3-совместимое хранилище, используемое для локальной разработки. В продакшен-среде его заменяет любое облачное S3-хранилище (Amazon S3, Yandex Object Storage и т.д.).

## Схема взаимодействия

1.  Клиент отправляет `POST` запрос с изображениями на эндпоинт `/v1/process` сервиса **`api`**.
2.  **`api`** создает `job_id`, сохраняет информацию о задаче в **`db`** (PostgreSQL) со статусом `queued`.
3.  **`api`** отправляет задачу на обработку изображений в очередь ( **`redis`** ).
4.  **`worker`** (Celery) забирает задачу из очереди. Статус задачи в **`db`** обновляется на `processing`.
5.  **`worker`** обрабатывает каждое изображение (изменяет размер, конвертирует) и загружает результат в **`minio`** (или другое S3-хранилище).
6.  После завершения обработки всех изображений **`worker`** обновляет запись о задаче в **`db`**, устанавливая финальный статус (`done`, `failed` или `partial_success`) и сохраняя URL-ы обработанных файлов.
7.  Клиент может периодически опрашивать эндпоинт `/v1/jobs/{job_id}` для получения статуса и результатов задачи.

Такая архитектура обеспечивает отказоустойчивость (если воркер упадет, задача останется в очереди) и масштабируемость (можно запустить несколько воркеров для обработки большего количества задач).
